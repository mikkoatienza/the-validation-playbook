You already have a problem worth solving, an offer, a price range, and a
channel that can reach buyers. Now you need proof that demand shows up when
you put it in front of real people.

This module is about testing, not building. You are looking for a clear
decision: do buyers act when the offer is real?

**Output for this module**
- An Experiment Card with a clear pass line
- A 7-day MVP test plan
- A pass/fail result with evidence notes

## MVP Means “Smallest Proof”
MVP usually stands for “minimum viable product.” In this book, treat it as
“minimum viable proof.” The goal is not a small product. The goal is a small
test that proves demand exists.

An MVP test isolates one assumption and makes it visible. If you cannot
name the assumption, you do not yet have a test.

Demand is not a compliment. Demand is action:
- A buyer books a call
- A buyer puts down money
- A buyer commits time to try the workflow

If the test does not ask for action, it cannot prove demand.

**What MVP is not**
- A mini version of the full product
- A long roadmap in disguise
- A brand launch or press push

**What MVP is**
- A small experiment that answers one question fast
- A test that forces a yes or no
- A learning step you can repeat next week

**Proof ladder (from weak to strong)**
- Interest: “I’d use this someday.”
- Intent: “I’ll take the next step.”
- Commitment: “I’ll pay or spend time now.”
- Repeat use: “I keep coming back.”

Your MVP should move at least one rung up this ladder.
Repeat use is a different test. You will measure that in the next module.

Example (generic):
If you think a clinic will pay for fewer no‑shows, do not build software.
Offer a manual reminder service for one week and see if they pay for it.

**Smallest proof checklist**
- Can it run in seven days?
- Does it require real buyers, not fans?
- Will the result force a decision?
- Can you repeat it next week if it works?

## The Experiment Card (Your Core Tool)
The Experiment Card keeps the test honest. It forces you to pick one
assumption, one test, and one pass line.

**Pick the riskiest assumption first**
Ask yourself:
- If this is false, does the idea die?
- Is this the least proven part of the idea?
- Can I test it in a week?

If the answer is yes, that is the assumption to test.

**Experiment Card fields**
- Assumption (the riskiest belief)
- Test (the smallest action that can prove it)
- ICP segment (who will see it)
- Metric (what you will count)
- Pass line (the number that says “yes”)
- Timebox (how long you will run it)
- Cost (money + hours)

**Assumption template**
"If [ICP] sees [offer], they will [action] because [reason]."

This makes the assumption visible and testable.

Example (generic):
- Assumption: clinic admins will pay to reduce no-shows
- Test: offer a paid pilot with a deposit
- ICP segment: clinics under 20 staff
- Metric: deposits collected
- Pass line: 3 deposits in 7 days
- Timebox: 7 days
- Cost: 10 hours + $0

**Metric rule**
Use one primary metric. If you need three metrics, the test is unfocused.

**Timebox rule**
If the test cannot fit in a week, shrink the scope until it can.

**Test design rules**
- Change one variable at a time
- Keep the offer and price stable
- Use the same message across attempts
- Decide your sample size before you start

If you feel tempted to add features, write them on a “later” list. The goal
is proof, not perfection.

show the concept. The video explained the problem and showed the workflow.
People signed up because they wanted the outcome, not the features.

That small test proved demand before the build caught up.

The video did not show a finished product. It showed a clear promise and
a simple workflow. That was enough to earn real sign‑ups.

## Pick a Test Type (Smallest Proof)
Use a test that matches your offer and channel. This is where founders
overbuild. Do not do that. Your job is to ask for action with the smallest
thing that makes the offer real.

- **Concierge MVP**: deliver the result manually to see if buyers will pay.
  This is slow, but it proves value fast.
- **Fake door test**: a button or link to “Get access” that leads to a waitlist.
  This tests intent before you build.
- **Paid pilot**: ask for a small deposit to try a limited version.
  Money is a strong signal.
- **Prototype walkthrough**: a click-through or demo with a clear next step.
  Use this to test understanding and commitment.
- **Problem walkthrough**: a 15‑minute call that ends with a paid next step.
  This tests urgency and budget.

Pick one. A clean test beats a mixed test.
If you are torn, choose the one that asks for money or time now.

**Mini examples (generic)**
- Concierge MVP: manually create a weekly report and charge for it.
- Fake door: “Start trial” leads to a waitlist with one qualifying question.
- Paid pilot: $200 deposit to reserve a two‑week pilot slot.
- Prototype walkthrough: show the flow and ask for a paid pilot at the end.

**When to use each test**
- Concierge MVP: when the workflow is manual and the outcome is clear
- Fake door: when you need to test pull before building
- Paid pilot: when you need proof people will pay at all
- Prototype walkthrough: when the workflow is new or hard to explain
- Problem walkthrough: when you need to test urgency and budget fast

**Match the test to the risk**
- If the risk is willingness to pay, use a paid pilot or deposit.
- If the risk is clarity, use a prototype walkthrough.
- If the risk is urgency, use a problem walkthrough with a firm next step.
- If the risk is distribution, use a fake door plus a real follow‑up.

Do not pick the test you like. Pick the test that answers the riskiest
question fast.

## Set the Pass Line (Do This First)
The pass line forces a decision. Without it, you will rationalize any result.

**Pass line rules**
- Tie it to action, not views
- Use a denominator (responses out of total attempts)
- Set it before the test starts
- Keep it realistic for a first run

Denominator = total attempts. Numerator = qualified responses.

Example (generic):
- 30 outreach messages
- Pass line = 5 qualified replies and 2 paid pilots

If you do not hit the line, it is a fail. Failing fast is useful evidence.

**Pass line sanity check**
Ask: "If I hit this, would I spend another week on this idea?"
If the answer is no, the pass line is too low.

**Confidence note**
After the test, add a 1‑5 confidence score. A single pass can still be a weak
signal if the channel was small or the buyer was an outlier. The score keeps
you honest about how much you really learned.

**Stop rules**
- If you hit the pass line early, stop and repeat the test.
- If you get zero signal after half the timebox, review the channel or offer.

**Denominator guidance**
If you can only reach 10 buyers this week, set the pass line around that.
Small tests are fine. What matters is the ratio and the clarity of the signal.

## The 7-Day MVP Sprint
Keep the test short so you can learn quickly.

Day 1: Write the Experiment Card and pass line  
Day 2: Build the smallest version (page, script, or prototype)  
Day 3: Launch to your chosen channel  
Day 4-5: Follow up and collect responses  
Day 6: Review results against the pass line  
Day 7: Decide: repeat, revise, or drop

Speed beats depth here. You are proving demand, not shipping.

Before Day 1, gather a list of real buyers. A test without distribution is
not a test.

**Follow‑up script (short)**
"Did you get a chance to look? If this solved [pain], would you want to try
it next week?"

Follow‑up is part of the test. Many true signals arrive after the reminder.

**Smallest build examples**
- A single landing page with one call‑to‑action
- A one‑page PDF that outlines the offer
- A short demo script you can reuse
- A clickable prototype with one primary flow

ICP. A test without real targets turns into guesswork.

**Decision note**

- If I pass, I will _______.
- If I fail, I will _______.

This keeps the test honest.

## Evidence Artifact (MVP Test Summary)
Capture your results in one place:

- Assumption tested
- Test run
- Attempts
- Qualified responses
- Commitments (time or money)
- Pass line result (pass/fail)
- Notes on objections

**MVP Test Summary (simple)**
| Item | Notes |

| Assumption | |
| Test | |
| Attempts | |
| Qualified responses | |
| Commitments | |
| Pass line | |
| Result | |
| Objections | |

This artifact becomes your evidence trail. It keeps you honest when the
next idea looks shiny.

**Evidence strength (quick scale)**
- Strong: money or time commitment from the buyer
- Medium: qualified reply + clear pain
- Weak: interest with no action

**Summary paragraph**
Write 3 to 5 sentences that say what happened, what surprised you, and what
you will test next. This is your founder memory.

**What to do after the result**
- If you pass: repeat the test and increase volume.
- If you fail: change one variable (offer, channel, or ICP) and rerun.

Do not change everything at once. You will not know what caused the shift.

## Common Traps
- Building a full product before any proof
- Measuring clicks instead of commitments
- Changing the offer mid-test
- Asking for feedback instead of action
- Testing with fans who will say yes to anything
- Setting a pass line after the results are in

If you recognize one of these, reset the test. It is better to lose a week
than lose six months.

If you catch yourself saying, “It’s too early to ask for money,” shrink the
ask instead of avoiding it. Ask for a deposit. Ask for a paid pilot. Ask for
an hour of their time. Demand shows up in small actions.

## Closing Line
If the test does not force a decision, it is not a test.

Next, you will check whether people come back and use it again. That is where
early fit shows up.
