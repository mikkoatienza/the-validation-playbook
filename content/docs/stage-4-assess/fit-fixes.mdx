You now know where fit is weak. The temptation is to change everything.
Resist that. Your job is to fix the blocker, not chase shiny changes.

This module helps you choose one path -- improve, pivot, or stop -- based on the
signals you just measured.

**Output for this module**
- A clear iteration path (improve, pivot, or stop)
- One focused change to test next
- A short evidence note that explains why

## Iteration Rules (Plain Language)
Iteration is not random change. It is a decision rule.

There are only three paths:
- **Improve**: the core is right, the experience is weak.
- **Pivot**: the core is wrong, but the learning is valuable.
- **Stop**: the evidence is clear and the cost is too high.

If you change everything, you will not know what worked. If you change
nothing, you will drift.

**Iteration vs experimentation**
An iteration is a targeted change to fix a known blocker.
An experiment is the test that proves the change worked.

You need both. Change without a test is guessing. Tests without a change are
busywork.

**How to read the signals**
- Improve when at least one strong signal exists (people return sometimes).
- Pivot when no strong signals exist but you see a consistent pattern of pain.
- Stop when signals are weak and the cost to learn more is too high.

Be honest about the cost. Time is a real expense.

**The “one‑variable” rule**
Pick one variable to change in the next test. If you change two, you will not
know which one helped. If you change three, you will confuse yourself.

**Variable examples**
- Offer promise
- Target segment (ICP)
- Price point
- Channel
- Time‑to‑value steps
- Core workflow step

**When to use each path (quick signals)**
- Improve: repeat use is low, but people clearly finish the core action.
- Pivot: completion is low and the pain does not feel urgent.
- Stop: repeat use is near zero after two focused fixes.

If you are unsure, start with improve. If that fails twice, consider pivot.

**Path decision questions**
- “Is the problem still painful for the people who tried it?”
- “Did anyone come back without me pushing them?”
- “If I fix the weakest signal, does the loop make sense?”

**Path examples (generic)**
- Improve: users finish the task, but only 2 of 10 return. Your fix is to
  reduce time‑to‑value so the first win happens sooner.
- Pivot: users do not finish the task, but they keep asking for a different
  outcome. Your pivot is to serve that outcome directly.
- Stop: after two focused fixes, repeat use is still near zero and the cost
  to keep testing is high. You stop and reuse what you learned elsewhere.

## The Three Paths (One‑Sentence Cues)
Use these cues to choose your path:

- **Improve** when repeat use exists but is inconsistent.
- **Pivot** when the loop is weak and the pain is not urgent.
- **Stop** when you cannot earn repeat use after focused fixes.

This is not a moral decision. It is an evidence decision.

## A Real Example (Slack’s Pivot)
Slack started as a game company. The game failed, but the team noticed their
internal chat tool worked better than anything else. They pivoted to focus on
the tool, not the game. The lesson is not “pivot often.” The lesson is “pivot
when the evidence shows a different product has pull.”

That is what a focused pivot looks like.

Slack did not add features to save the game. They focused on the tool that
already had pull.

The pivot was not random. It was a response to clear usage patterns.

Slack’s team noticed which tool people would not give up. That was the clue.

They did not rename the game and hope. They followed the evidence.

**Signal to watch**
If a small feature keeps getting reused while the core product fades, you may
be looking at your real product.

That is the pivot test: repeat use rarely lies.

**Pivot guardrails**
Pivot does not mean burning everything down. Keep what still works:
- Keep the channel if it reaches the right people
- Keep the strongest message that earned replies
- Change the outcome or workflow that blocked repeat use

A good pivot preserves evidence and drops the dead weight.

## The Iteration Filter (Pick One Change)
Use this filter to avoid thrashing:

1. **Name the blocker**: which signal was weakest?
2. **Name the cause**: why is that signal weak?
3. **Name one change**: what single change targets the cause?

If you cannot name one change, you likely do not understand the blocker yet.

**Blocker examples**
- Repeat use is low because the value shows up too late.
- Completion is low because the workflow is unclear.
- Reactivation is low because the trigger is weak.

Name the blocker in plain language before you solve it.

**Cause prompts**
- “People drop because ___.”
- “The loop breaks after ___.”
- “The reason to return is unclear because ___.”

**Change types (pick one)**
- Cut steps to reach value (shorter time‑to‑value)
- Clarify the core promise (better understanding)
- Tighten the ICP (right people, right pain)
- Raise or lower the ask (fit to budget/urgency)
- Adjust the loop trigger (better return timing)

If you cannot link the change to the weakest signal, it is probably a shiny
idea, not a fix.

**Example (generic)**
- Weakest signal: repeat use
- Cause: value shows up after week two
- Change: deliver a quick win in day one
- Test: 10 users, measure week‑one return
- Pass line: 3 of 10 return in the expected cycle

**Iteration grid (simple)**
| Weakest signal | Likely cause | One change | Next test |
| --- | --- | --- | --- |
| | | | |

Use this grid to force one choice and one test.

**Two‑cycle check**
Run two focused tests before you decide to pivot or stop. One test can be a
fluke. Two tests show a trend.

## Exercise: Choose Your Path + One Change
Complete this in one sitting:

**Path decision**
- My weakest signal is:
- The cause is:
- I will: Improve / Pivot / Stop
- My one change is:
- Why this change:

Keep the change small enough to test in 7 days.

**Timebox rule**
If you cannot test it in a week, the change is too big.

**Message constraint**
Write the change in 12 words or fewer. If you cannot, it is not focused yet.

**Next‑experiment prompt**
Write the exact test in one line: “We will change ___ and measure ___ in 7 days.”

If you can’t write that line, you don’t have a test yet.

**One‑sentence rationale**
Write: “This change targets ___, which is our weakest signal.”

**Sample size note**
Write the number of people in the test. If it is small, acknowledge it.

**Evidence reminder**
Keep the notes from the last module open while you write this plan. If the
evidence is not on the table, you will drift back to opinion.

**Pre‑mortem (2 minutes)**
Write one line: “This change fails because ___.”
If the reason is unclear, the test is too fuzzy.

After the test, compare the result to the pass line and update the weakest
signal. If the weakest signal did not move, change the variable, not the
volume.

## Evidence Artifact (Iteration Decision Note)
Capture your decision in a short note:

- Weakest signal
- Chosen path (improve/pivot/stop)
- One change to test
- Next experiment

**Iteration Note (simple)**
| Item | Notes |
| --- | --- |
| Weakest signal | |
| Path | |
| One change | |
| Next experiment | |

This prevents you from rewriting history after the next test.

**Decision line**
Write one sentence: “We are improving/pivoting/stopping because ___.”
If you can’t say it cleanly, you are not ready to move.

**If you choose “stop”**
Write one sentence about what you will reuse:
- The segment you learned
- The channel that worked
- The offer language that got replies

Stopping is not failure. It is focus.

**Carry‑forward note**
Write one sentence: “What I learned here will help me because ___.”

Stopping is only a loss if you forget the evidence.

**Pass line note**
Add a pass line for the next experiment so you do not shift the goalpost.

**Cost note**
Write the expected time and cash cost. If it is too high for the signal,
you are over‑investing.

**Energy note**
Write one line on your own energy. If the evidence is weak and the energy is
gone, that is a real signal.

## Common Traps
- Changing three variables at once
- Chasing opinions instead of signals
- Pivoting because you are bored
- Avoiding the stop decision
- Confusing “more effort” with “better evidence”

If you cannot explain your change in one sentence, it is not focused.

**Warning sign**
If your plan includes “launch and see,” you are skipping the rule. Pick a
pass line, then decide.

**Momentum check**
If you are exhausted and still have no repeat use, stop is a valid choice.

**Decision clarity**
If you cannot say “improve,” “pivot,” or “stop” without hedging, you are
avoiding the decision. That is a signal too.

**External noise check**
Competitor news or social hype is not a signal. Your signal is repeat use.
If nothing changed in your data, do not change the plan.

If you keep postponing the decision, set a date and decide.

Clarity beats comfort. Evidence beats ego. Decide now.

## Closing Line
Fix the blocker or walk away.
