Momentum is not a burst. It is a loop you can repeat.
Retention comes before growth. Evidence comes before scale.
This extension is about building the loop that keeps your proof alive.

You already earned commitment and early fit. Now you need momentum that does
not depend on hype or heroics. That means choosing metrics that map to real
behavior, strengthening the retention loop, and adding growth only when the
loop is strong enough to carry it.

This chapter is case-study anchored, but the goal is not to copy a company.
The goal is to see how evidence turns into a loop, and how a loop becomes
momentum you can run every week.

## Section A: Metrics That Matter (After Proof)
The metrics you use after validation are not the same metrics you used to
validate. At this point, you are not proving interest. You are proving
repeatable behavior.

**Signal metrics** reflect behaviors that indicate value is real.
**Vanity metrics** are numbers that can move without changing behavior.

Signal metrics look like:
- Repeat use within the natural problem cycle.
- Time-to-value shrinking over time.
- A return trigger that works without manual nudges.

Vanity metrics look like:
- Page views without follow-up action.
- Sign-ups without return behavior.
- Social spikes without repeat usage.

### Lagging vs Leading Signals
Signal metrics can still be too slow. A lagging signal tells you what already
happened. A leading signal tells you what is about to happen. You need both,
but you must know which is which.

Examples:
- **Lagging:** weekly active usage, paid renewals, repeat purchases.
- **Leading:** time-to-value, completion of the first meaningful action,
  return within the first cycle.

If you only measure lagging signals, you will learn too late. If you only
measure leading signals, you may celebrate a loop that never holds.

### The Case Thread (Figma)
Figma's early momentum was built around collaboration. Design work naturally
repeats, and the product made shared files the default. That meant the signal
metric was not "sign-ups." The signal was "teams returning to the same file,
week after week." That behavior tied directly to value.

The exact numbers do not matter here. The lesson does: they tracked behavior
that proved the product was part of a repeatable workflow.

**Case takeaway:** choose a signal that shows repeat collaboration, not just
interest. The metric should prove the loop is alive.

### Metric Pairing (One Outcome, Two Behaviors)
Pick one outcome and two behaviors that predict it.
Example:
- **Outcome:** weekly retained teams.
- **Behavior 1:** first collaboration within 24 hours.
- **Behavior 2:** second session within 7 days.

If the behaviors do not move, the outcome will not move. That is the point.

### The Metric Triad
If you want a simple model, use this triad:
1. **Activation:** how quickly users reach the first real value.
2. **Retention:** how often they return within the expected cycle.
3. **Expansion:** how the loop grows through usage (invites, sharing, or more
   usage per user).

If you cannot name these three for your product, you are not ready to talk
about growth.

### Example North-Star Signals (By Product Type)
Use these as inspiration, not rules:
- **B2B workflow tool:** weekly active teams using the core workflow.
- **Marketplace:** completed transactions per week.
- **Consumer habit app:** weekly returning users within the problem cycle.
- **Collaboration product:** files with two or more active contributors.

The right signal is the behavior that proves value, not the behavior that is
easy to count.

### Mini-Activity: Signal Metrics Pick
Choose one north-star metric and two supporting signals that map to behavior.

**Signal Metrics Pick**
- **North-star signal:** (the behavior that proves value)
- **Support signal 1:** (leading indicator)
- **Support signal 2:** (leading indicator)
- **Pass line:** (what counts as "good enough" this month)

If you cannot set a pass line, you cannot decide. If you cannot link the
metric to behavior, it is not a signal.

### Metric Definition Card (Keep It Clean)
Write the definition for each metric you track:
- **Event:** the exact action that counts.
- **Actor:** which user or account counts.
- **Window:** the time window that qualifies.
- **Pass line:** what "good enough" means this month.

If your team cannot repeat the definition, the metric will drift.

### A Short Reality Check
Ask yourself:
- "If this metric doubles, would the business be healthier?"
- "Does this metric represent real customer behavior?"
- "Can I influence this metric within 30 days?"

If any answer is no, you are tracking the wrong thing.

### Metric Hygiene (Avoid Noise)
Three simple rules keep your metrics clean:
- **Measure by cohort, not just totals.** Totals hide churn.
- **Name the behavior, not the tool.** "Weekly report created" beats "active."
- **Avoid mixing segments.** One segment can mask another.

If your metric needs a footnote, it is probably not a signal.

Avoid these traps:
- **Metric swapping:** changing the metric when the signal is weak.
- **Metric stacking:** tracking too many signals and deciding nothing.
- **Metric nostalgia:** sticking to a metric because it used to look good.

Every metric should lead to a decision line:
"If [signal] is above [pass line], we will ___. If it is below, we will ___."

If you cannot write the decision line, you are not ready to track the metric.

### Metric Ownership and Cadence
Metrics only matter if someone owns them. Choose one owner per signal and one
weekly review slot. If the review does not lead to a decision, it is theater.

Use a simple cadence:
- **Weekly:** review signal change and decide the next test.
- **Monthly:** revisit the pass line and adjust based on evidence.

If you cannot name the owner or the review time, your metrics will drift.

Use a simple sequence: set the Signal Metrics Pick and Metric Definition Card
once, review weekly, revisit the pass line monthly.

### Metrics Must Match the Edge
Your positioning and edge should show up in the metrics you track. If you say
you win on speed, measure time-to-value. If you say you win on reliability,
measure repeat use without support. If you say you win on access, measure how
often buyers return through the chosen path.

If the metric does not reflect the edge, the edge is not real yet.

## Section B: Retention and the Core Loop
Retention is the heartbeat of momentum. If buyers do not return, growth just
creates a bigger leak. The core loop is what brings people back without you
chasing them.

The simplest loop looks like this:
**Trigger -> Action -> Reward -> Return**

The loop breaks when:
- The trigger is weak or inconsistent.
- The action is too hard or too slow.
- The reward is unclear or delayed.
- The return is optional, not needed.

### Common Loop Breakers
Most weak loops fail for one of these reasons:
- **Trigger mismatch:** the product does not align with a real routine.
- **Friction overload:** too many steps before value.
- **Reward ambiguity:** users do not feel the benefit quickly.
- **Return gap:** no natural reason to come back.

If you fix the wrong break, you keep the loop weak.

### Loop Break Diagnostic
Use this quick diagnostic to find the real break:
- **Trigger missing:** users forget because the trigger is external.
- **Action friction:** too many steps before value.
- **Reward unclear:** the outcome is not obvious or not immediate.
- **Return optional:** the loop does not tie to a real need.

Name the break, then design one change to address it. Do not fix all four.

### The Case Thread (Figma)
In Figma's early loop, the trigger was the work itself. A team needed to
design, review, and update files regularly. The action was easy: open the
shared file, make changes, and comment. The reward was fast feedback. The
return happened naturally because the project continued.

This loop is powerful because the trigger is real and recurring. The product
did not have to invent a reason to return. It reinforced one that already
existed.

Early on, collaboration tools like this often try to grow before the loop is
stable. When invites do not convert or usage stalls after the first session,
it is a signal to fix the loop, not to push harder on acquisition. The pattern
is consistent: once the core loop works, sharing becomes a multiplier instead
of a leak.

**Case takeaway:** retention comes from a real trigger. Growth only works once
the trigger is strong and repeatable.

### The Retention Curve Lens
You do not need complex analytics to read a retention curve. You need the
shape:
- **Sharp drop with no return:** the loop is weak.
- **Drop then flatten:** a core group is sticking.
- **Gradual decline:** the loop is fragile but improving.

If you cannot see a shape, you probably do not have a loop yet.

### Segment the Curve
Average retention hides truth. Segment by:
- Buyer role (decision maker vs user).
- Use case (primary workflow vs edge case).
- Time-to-value (fast vs slow starters).

If one segment retains and another churns, the loop may be real for one group.

### The Frequency Fit Check
A loop that returns weekly is different from a loop that returns monthly. If
you expect weekly returns but the problem only appears monthly, you will
interpret normal behavior as churn.

Match the loop frequency to the problem cycle. If you cannot name the cycle,
you cannot judge retention fairly.

### Mini-Activity: Loop Map
Sketch your loop in plain language.

**Loop Map**
- **Trigger:** what starts the loop?
- **Action:** what the user does next.
- **Reward:** what makes it feel worth it?
- **Return:** why they come back.
- **Blocker:** what breaks the loop?

Then write one sentence:
"The loop breaks when ___." Fix that first.

### Evidence Tests for Loop Strength
Use these tests to validate the loop:
- **Frequency:** does the user return within the expected cycle?
- **Reactivation:** do they return after a gap without heavy prompting?
- **Consistency:** does the loop work for a segment, not just one user?

If you cannot pass these tests, your loop is still a hypothesis.

### Return Triggers vs Reminders
Reminders are not triggers. A trigger is a real reason to return.
If users only come back after a reminder, your loop is still weak.
Use reminders to learn, not to pretend the loop is strong.

### Reactivation Without Panic
Reactivation is a test, not a rescue. If users return only after repeated
nudges, the loop is weak. If they return after one clear reminder tied to a
real trigger, the loop is improving.

Use a simple reactivation test:
- One reminder at the expected cycle.
- One reminder after a clear lapse.
- Compare return behavior to the baseline.

If reactivation requires a long sequence, the loop needs to change.

### The Time-to-Value Rule
Time-to-value is the gap between "first contact" and "first real benefit."
If that gap is long, retention will be weak. Shrink the gap before you
optimize anything else.

Ask: "What is the smallest moment of real value, and how fast can we deliver
it?" That answer is your first retention project.

### Time-to-Value Experiments
Shorten the path to first value with one change at a time:
- Remove one step from onboarding.
- Pre-fill the first action.
- Offer a guided example instead of a blank state.

If time-to-value does not shrink, retention rarely improves.

### The Retention Three-Step
Retention improves when you focus on three steps:
- **Activation:** make the first value moment obvious.
- **Habit:** make the loop easy to repeat.
- **Reactivation:** bring people back after a break.

If you only optimize activation, you get a spike. If you only optimize
reactivation, you mask a weak loop. Work in order.

## Section C: Growth Loops Without Hype
Growth comes from loops, not campaigns. If retention is weak, growth is just
expensive churn. The order matters: **Retention first, then growth.**

Growth loops are built on usage:
- A user shares a file that invites another user.
- A collaboration creates new use cases inside the product.
- A data loop makes the product smarter, which attracts more usage.

If the loop is not anchored in behavior, it will not repeat.

### Loop Math (Keep It Simple)
Every growth loop has three moving parts:
- **Input:** a user or action that starts the loop.
- **Output:** a new user or new usage that returns to the input.
- **Cycle time:** how long it takes to complete one loop.

You do not need precise math to start. You do need to know if the output is
larger than the input over a reasonable cycle. If it is not, the loop will
stall.

### Loop Speed Matters
A slow loop can still work, but it must be realistic. If your loop takes
months to complete, you will learn slowly and burn cash. Aim for the fastest
credible cycle you can test.

Ask: "How quickly can one loop complete, and what would make it 20 percent
faster?" Speed makes learning possible.

### The Case Thread (Figma)
Once the core collaboration loop was working, growth became a byproduct.
Sharing a file brought in new collaborators. Those collaborators became
regular users because the workflow demanded it. The loop grew because the
product made collaboration the default, not an add-on.

The lesson is not "add sharing." The lesson is "grow from the behavior that
already repeats."

Early attempts to push growth before the loop is stable often fail. When
collaboration tools promote sharing too early, invites can turn into one-time
visits. The adjustment is simple but hard: fix the core loop first, then
reintroduce the growth trigger. When the loop is real, the same trigger works.

**Case takeaway:** a growth loop is a layer on top of retention, not a
replacement for it.

### Growth Loop Types (Simple)
Pick the growth loop that fits your product:
- **Collaboration loop:** users invite others to work together.
- **Referral loop:** users share because it helps them.
- **Content loop:** users create assets that attract more users.
- **Data loop:** usage improves outcomes, which attracts more usage.

Choose one. If you try to run all of them, none of them will compound.

### Loop Stacking (Only After Retention)
Once a retention loop is stable, you can stack a growth loop on top of it.
If you stack before retention is stable, you magnify churn.
The sequence matters: retain, then expand.

### Growth Loops That Are Not Loops
If the loop does not feed itself, it is not a loop:
- A campaign spike that ends when the spend stops.
- A referral program that works once but does not repeat.
- A content burst without return behavior.

If the output does not become a new input, you are not compounding.

### Mini-Activity: Loop Experiment Card
Design a small test to validate a growth loop.

**Loop Experiment Card**
- **Hypothesis:** "If we add ___, then ___ will happen because ___."
- **Trigger:** what starts the loop?
- **Action:** what the user does.
- **Reward:** what makes it worth repeating.
- **Pass line:** what signal proves the loop is alive?
- **Timebox:** how long you will run the test.

If you cannot set a pass line, it is not a loop test. It is a hope test.

### Loop Experiment Pitfalls
Avoid these common mistakes:
- Testing a loop without a clear trigger.
- Measuring volume without measuring repeat behavior.
- Running the test without a timebox.

If you cannot define the end of the test, you cannot learn from it.

### Loop Readiness Checklist
Run this before you chase growth:
- Retention signal is strong and repeatable.
- Trigger is consistent and clear.
- Time-to-value is short enough to repeat.
- One loop test can be run in 30 days.

If you cannot check these boxes, delay growth. Fix the loop first.

### Loop Ownership
Growth does not belong to marketing alone. A loop touches product, onboarding,
and behavior. Assign one owner who can change the loop, not just report it.
If ownership is unclear, growth experiments become status updates instead of
decisions.

### The Weekly Loop Review
Momentum improves when you review the loop weekly:
- What signal improved?
- What broke the loop?
- What is the next smallest test?

If you skip the review, you skip the compounding.

### Mini-Activity: Weekly Loop Review
Write a short weekly review using this template:

**Weekly Loop Review**
- **Signal move:** what changed this week?
- **Loop break:** where did the loop fail?
- **Next fix:** the smallest test for the next week.
- **Decision:** keep, adjust, or pause the loop test.

If you cannot write the decision, you are collecting data without a plan.

Use the Loop Readiness Checklist before growth tests, the Weekly Loop Review
every week, and the Momentum Scorecard once a month.

## The Case Thread (What It Teaches)
The Figma thread is not a template. It is a reminder that loops are built from
behavior, not from slogans. Collaboration created retention. Retention created
growth. The loop did the work, not a campaign.

You do not need that exact loop. You need a loop you can explain, test, and
repeat.

## Momentum Without Burnout
Momentum is only useful if it is sustainable. If your loop requires a heroic
push every week, it will break. The goal is a loop you can run at a steady
pace with the team and resources you already have.

Use this rule of thumb: if you cannot run the loop for four weeks in a row
without slipping, the loop is too heavy. Shrink the test, reduce the scope,
or tighten the trigger until it fits your capacity. A smaller loop that
repeats beats a larger loop that collapses.

## Momentum Scorecard (Monthly)
Once a month, score the health of the loop. Keep it simple:
- **Signal metric:** is the core behavior improving?
- **Retention shape:** is the curve flattening for a real segment?
- **Loop clarity:** can you explain trigger -> action -> reward -> return?
- **Growth test:** is one loop test above the pass line?
- **Cadence:** did you run the weekly review four times this month?

If two or more scores are weak, slow down and tighten the loop before you add
new growth tests. Momentum debt compounds fast.

## Momentum Debt (Short Note)
When you skip the loop review, ignore weak signals, or chase growth before
retention, you create momentum debt. The debt shows up later as churn, stalled
loops, and loss of focus. The fastest way to pay it down is to return to the
core loop and fix the smallest break.

Momentum is a decision repeated weekly. If you cannot repeat it, you have not
found the right loop yet. Reduce the scope until the decision becomes easy to
keep.
