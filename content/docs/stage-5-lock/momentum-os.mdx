Commitment without a system fades fast. The real danger is not a bad idea.
It is a good idea that drifts because the founder has no weekly rhythm.

This module installs a simple operating system: one weekly evidence sprint.
It keeps you focused on proof, not noise. It is how you protect momentum.

**Output for this module**
- A weekly evidence sprint schedule
- A written evidence log template
- The next four sprints on your calendar

## What a Momentum OS Is (Plain Language)
Momentum OS is a repeatable weekly loop that forces learning. It is not a
product roadmap. It is a rhythm that asks, every week:

- What is the riskiest assumption right now?
- What is the smallest test that can answer it?
- What did we learn, and what do we do next?

If you skip the loop, the business runs on hope and adrenaline. If you run the
loop, the business runs on evidence.

Momentum is not a mood. It is a system.
You can feel inspired and still fail. The system turns inspiration into
evidence and evidence into action.

## Why Weekly Works
Weekly is short enough to feel urgent and long enough to learn. Daily is too
noisy. Monthly is too slow. A week gives you a clean rhythm and a clean reset.

If you do not decide weekly, you carry decision debt. It piles up and makes
everything heavier.

## The Weekly Evidence Sprint (Choose, Run, Review)
Every week has three moves. Keep them small and repeatable.

**1) Choose (Monday)**
- Pick the single riskiest assumption.
- Write the pass line before you test.
- Define one metric you will watch.

If you cannot name the assumption in one sentence, you may not be ready to test.

**2) Run (Tuesday–Thursday)**
- Execute the smallest test.
- Keep the offer, price, and channel stable.
- Track the single metric daily.

**3) Review (Friday)**
- Compare results to the pass line.
- Decide: repeat, refine, or stop.
- Write the one‑sentence learning.

That is the whole system. If you can do that every week, you are far less
likely to drift.

Diagram cue: weekly loop (Choose → Run → Review → Choose).

**Pass line rule**
If the pass line feels too easy, raise it. If it feels impossible, shrink the
test. The pass line should make the decision obvious.

**Leading vs lagging signals**
Leading signals tell you if the loop is working now (responses, returns,
time‑to‑value). Lagging signals show up later (revenue, retention over months).
Weekly sprints are for leading signals. Don’t wait for revenue to learn.

## The Weekly Review Script (10 Minutes)
Keep the review short so you actually do it.

Ask:
- What did we test?
- Did we hit the pass line?
- What did we learn in one sentence?
- What is the next riskiest assumption?

If you cannot answer in 10 minutes, the test was too big.

**Review example (generic)**
- Test: 20 outreach messages to clinics
- Pass line: 5 qualified replies
- Result: 4 replies
- Learning: pitch works, channel needs volume
- Decision: repeat with 40 messages next week

## A Real Example (Amazon’s Weekly Business Review)
Amazon is known for its weekly business review (WBR). The company uses a
structured weekly review to focus on leading indicators and decide what to do
next. It is not a “status update.” It is a decision rhythm.

The lesson is not “be Amazon.” The lesson is “build a weekly cadence that
forces honest decisions.” The system makes momentum repeatable.

You do not need Amazon’s scale. You need a weekly decision ritual that keeps
you honest about what the evidence says.

## The Three Decisions (Repeat, Refine, Stop)
Every review ends with one of three calls:

- **Repeat**: the test worked; run it again with more volume.
- **Refine**: the signal was weak; change one variable and retest.
- **Stop**: the signal failed; choose a new assumption.

These decisions keep the system alive. Without them, you just collect notes.

## The Evidence Log (Founder Memory)
Without a log, you will repeat the same mistakes. The log is not a journal.
It is a record of tests, results, and decisions.

**Evidence log fields**
- Assumption
- Test
- Pass line
- Result
- Learning
- Next test

If you cannot write the learning in one sentence, the test was too fuzzy.

The log gives you the story of the work. The next tool gives you the signal.

## The Weekly Scoreboard (One Line)
Each week, write a one‑line scoreboard. This is the fastest way to see if you
are making progress:

**Scoreboard line**
“We tested ___, expected ___, got ___, learned ___, next ___.”

If you skip this line, the week disappears into memory fog.

Once you can see the signal, you can separate what you control from what you
hope for. That is where metrics help.

## Input vs Output Metrics (Keep It Simple)
Outputs lag. Inputs show you what you control today.

**Input examples**
- Outreach attempts
- Demos run
- Pilot offers made

**Output examples**
- Paid pilots
- Repeat use
- Renewals

Track both, but decide based on outputs. Inputs only tell you effort.

## Sprint Rules (Keep It Honest)
Use these guardrails so the system doesn’t drift.

- **One test per week.** More tests means less learning.
- **One pass line.** No pass line means no decision.
- **One metric.** More metrics means excuses.
- **One change at a time.** If you change two variables, you learn nothing.

The system is simple on purpose. Complexity hides failure.

If you miss a week, restart the loop immediately. Do not “catch up” by
stacking two tests. One clean test beats two rushed ones.

**Minimal test checklist**
- Can it run in seven days?
- Does it ask for real action?
- Is the pass line written?
- Is the metric clear?

If any answer is no, shrink the test.

## The Loop Integrity Check
At the end of each week, answer these questions:

- Did we test the riskiest assumption?
- Did we hit the pass line?
- Did we make a decision?
- Did we schedule the next test?

If any answer is “no,” the loop is broken. Fix it before you build more.

**Loop reset**
If the loop breaks, do a 30‑minute reset:
- Re‑state the riskiest assumption
- Set a pass line
- Schedule the next review

**Reset question**
Ask: “What would make next week a clear yes or no?” If you cannot answer,
the test is still too big.

**Learning note**
Write one sentence: “We learned ___, which changes ___.” If you can’t write
it, you don’t yet know what happened.

- Evidence log with pass lines
- Review ritual (day/time)

**Weekly cadence note**
| Item | Notes |
| --- | --- |
| Review day/time | |
| Evidence log location | |
| Next 4 sprints scheduled | |

This artifact turns your commitment into a repeatable system.

**Cadence rule**
Choose a fixed review time (Friday afternoon works for many founders). If the
time floats, the review disappears.

**Accountability check**
Write who will see the evidence log. If no one sees it, it is easy to ignore.

**Solo founder note**
If you are alone, send the log to a trusted friend or advisor. The point is
not approval. The point is discipline. A log with an audience changes how you
show up.

**Friday review checklist**
- Pass line written
- Result recorded
- Decision made
- Next sprint scheduled

If you skip one, fix it before Monday.

## Common Traps
- Skipping the review because you are “too busy”
- Running multiple tests and learning nothing
- Changing the offer mid‑test
- Tracking vanity metrics instead of decisions
- Ignoring weak signals because one week felt good

Momentum often dies when the loop breaks. Protect the loop.
If you stop logging, you stop learning. The log is the system.

Good weeks still need reviews. That is when you learn what worked.
Bad weeks need reviews even more. That is how you correct course.

If the week ends without a decision, the sprint didn’t happen. The decision
is the product of the sprint.

## Closing Line
No system, no momentum.
